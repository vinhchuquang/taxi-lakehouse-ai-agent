# Evaluation Methodology

This document formalises the evaluation protocol behind the agent's 27-case
regression harness and the data-quality and performance reports. It is intended
to back Chapter 5 (Evaluation) of the thesis report.

- Defense dataset window: `2024-01-01` through `2024-06-30` (frozen).
- Latest evaluation snapshot: [agent-evaluation-results.json](../agent-evaluation-results.json).
- Performance benchmark: [performance-report.md](../performance-report.md),
  [performance-benchmark-results.json](../performance-benchmark-results.json).

---

## 1. Goals of evaluation

The evaluation answers three thesis-level questions:

1. **Correctness** — Does the agent return factually correct answers for
   in-scope analytical questions over Gold data?
2. **Safety** — Does the agent reject unsafe requests (DDL/DML, Silver/Bronze
   access, wildcard misuse, invalid joins) before any query executes?
3. **Performance** — Are end-to-end latencies acceptable for an interactive
   demo on local hardware?

The methodology deliberately mirrors the three guardrail layers documented in
[AGENTS.md](../../AGENTS.md): column, table, and join.

---

## 2. Evaluation framework

### 2.1 Reproducibility constraints

- All evaluation runs use the **same defense dataset** (`2024-H1`) generated by
  the dbt pipeline at the verified run `phase25_2024_01_20260506`.
- The DuckDB warehouse file is rebuilt from scratch through Airflow before each
  evaluation snapshot.
- The harness is executed inside the `api` Docker container so the Python
  runtime, `sqlglot`, and `duckdb` versions are pinned by the image.
- The semantic catalog ([contracts/semantic_catalog.yaml](../../contracts/semantic_catalog.yaml))
  must be unchanged between rebuilds; any change requires re-running the full
  27-case suite.

### 2.2 Test set design

The harness contains **27 cases** in three categories:

| Category | Count | Definition |
|---|---|---|
| answer | 13 | Question must return a deterministic answer grounded in executed SQL rows. |
| clarification | 3 | Question is ambiguous (missing grain/metric/time) — agent must request clarification rather than guess. |
| blocked | 11 | Question is unsafe or out-of-scope — guardrails must reject before execution. |

Within `answer`, cases are split by query surface:

- **aggregate_mart** (6 cases) — questions answerable from
  `gold_daily_kpis` or `gold_zone_demand` (fast path).
- **star_schema** (7 cases) — questions that require `fact_trips` + dimensions
  with controlled joins.

Within `blocked`, each case targets one guardrail axis:

- DDL / DML (CREATE, INSERT, UPDATE, DELETE) — statement-type rejection
- Bronze / Silver table access — table catalog rejection
- Unknown column — column guardrail rejection
- `SELECT *` on `fact_trips` — wildcard rejection
- JOIN without `ON` — join guardrail rejection
- Cartesian JOIN — join guardrail rejection
- JOIN path not in `allowed_joins` — join guardrail rejection

This produces broad coverage across the three guardrail layers and the two
serving surfaces.

### 2.3 Case selection bias — explicit declaration

Cases were designed by the author from the demo scenarios in
[demo-scenarios.md](../demo-scenarios.md) and prior anomaly tickets in the
pipeline metadata. This is **selection bias**: the cases reflect the questions
the author expected demo users to ask, not a uniform sample over question space.

This is consistent with thesis-scope evaluation for a defended MVP, but should
be reported transparently in Chapter 5.5 (Threats to validity).

## 3. Metrics

### 3.1 Correctness

| Metric | Definition | Target |
|---|---|---|
| `successful_answer_pass_rate` | fraction of `answer` cases that return the expected row count and grounded answer | 1.0 |
| `clarification_pass_rate` | fraction of `clarification` cases that return a clarification prompt (HTTP 200 with `requires_clarification = true`) | 1.0 |
| `grounded_answer_rate` | fraction of returned answers whose natural-language text references only executed result rows | 1.0 |
| `trace_complete_rate` | fraction of responses containing a complete `agent_steps` trace | 1.0 |

### 3.2 Safety

| Metric | Definition | Target |
|---|---|---|
| `unsafe_rejection_rate` | fraction of `blocked` cases rejected before DuckDB execution | 1.0 |
| Per-guardrail rejection coverage | each guardrail axis must have at least one `blocked` case that exercises it | 100% |

### 3.3 Performance

| Metric | Definition | Target |
|---|---|---|
| `overall_p50` / `overall_p95` | end-to-end latency over all 27 cases | informational |
| `answer_p50` / `answer_p95` | latency over `answer` cases only | < 3000ms p95 |
| Latency by surface | `aggregate_mart` and `star_schema` reported separately | informational |

## 5. Data quality evaluation

Source: [data-quality-report.md](../data-quality-report.md),
[dbt/models/schema.yml](../../dbt/models/schema.yml).

- **dbt tests**: 77 PASS / 2 WARN / 0 ERROR / 0 SKIP.
- **Warning tests** are explicit anomalies, not failures:
  - `warn_silver_trip_anomalies` (≈18K rows) — TLC publishes rows with pickup
    dates outside the file's partition month; flagged but retained.
  - `warn_gold_metric_anomalies` (1 row) — single outlier in the daily KPI
    aggregate.
- Silver applies **validity filters**: pickup inside partition month,
  dropoff > pickup, amount > 0, distance plausible. Approximately 18M rows are
  filtered out across the full window, documented in
  [data-quality-report.md](../data-quality-report.md).

## 6. Performance evaluation

Source: [performance-report.md](../performance-report.md).

Five representative queries benchmarked over `2024-H1`:

| Query | Median latency | Rows returned |
|---|---|---|
| Daily KPI from `gold_daily_kpis` | 962 ms | 1,642 |
| Zone demand from `gold_zone_demand` | 1,265 ms | 60,901 |
| Vendor revenue from star schema | 3,701 ms | 3 |
| Payment mix from star schema | 4,062 ms | 6 |
| Top pickup→dropoff pairs from star schema | 1,078 ms | 50 |

**Conclusion**: keep Gold as DuckDB views (no materialised persistence) — the
gap between aggregate-mart and star-schema queries is acceptable for the demo
window and worth preserving for engineering simplicity.

## 7. Limitations and threats to validity

Disclose all of these in Chapter 5.5 of the report.

1. **Selection bias.** The 27 cases were authored from demo scenarios, not
   sampled from a held-out user log.
2. **No external ground truth.** Answer correctness is verified against
   deterministic SQL results, not against human-labelled expected answers from
   an independent annotator.
3. **Single-turn only.** The agent has no conversational state. Multi-turn
   evaluation is out of scope.
4. **Narrow dataset window.** `2024-H1` is 6 months. Year-on-year and seasonal
   effects are not evaluated.
5. **Yellow + Green only.** FHV / HVFHV are out of scope, so evaluation does
   not cover ride-hailing patterns.
6. **Local hardware.** Latency measurements are taken on the author's
   development machine inside Docker. Numbers will differ on production
   hardware and under concurrent load.
7. **OpenAI synthesis not evaluated separately.** When the answer-synthesis
   path is enabled, the natural-language wording depends on the LLM provider;
   only the *grounded-on-rows* property is validated, not stylistic
   correctness.

## 8. How to reproduce

1. `docker compose up -d` — bring the full stack online.
2. Trigger or backfill the DAG so the warehouse contains `2024-01` through
   `2024-06`. Reference command in [runbook.md](../runbook.md).
3. Verify dbt produced 77 PASS / 2 WARN / 0 ERROR via
   `check_pipeline_run.py`.
4. Run the agent evaluation harness inside the `api` container — the script
   writes `docs/agent-evaluation-results.json` and prints the metrics table.
5. Compare to the targets in Section 3. Any deviation must be investigated
   before claiming a passing snapshot.
